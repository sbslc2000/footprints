# S3
S3란 Simple Storage Service의 약자로, 안전하고 가변적인 저장공간을 제공한다.

## S3은 파일시스템이 아니다!
S3는 파일시스템처럼 동작하는 것처럼 보일 수 있지만, 근본적으로 객체 스토리지이다. 가장 큰 차이점은 **bandwidth**와 **병렬성**에 있다.

파일시스템은 일반적으로 하나의 디스크에 제한된 bandwidth를 가지며, I/O 작업이 디스크의 물리적 속도에 의존한다. 반면 S3는 분산된 스토리지 시스템으로, 데이터를 병렬적으로 업로드하거나 다운로드하는 것이 가능하다. 이를 통해 대규모 데이터를 효율적으로 처리할 수 있다.

이러한 특징 때문에 S3는 **대규모 데이터 전송**이나 **병렬 처리가 필요한 환경**에서 강점을 가진다. 파일시스템처럼 작은 파일을 자주 읽고 쓰는 용도로 사용하기보다는, 대용량 데이터를 병렬로 처리하거나 장기 보관용으로 사용하는 것이 적합하다.

## 저장 중 암호화 (Server-Side Encryption)
S3는 저장되는 모든 객체를 암호화하여 디스크가 물리적으로 탈취되더라도 데이터를 보호한다. 모든 방식이 봉투 암호화(Envelope Encryption) 구조를 사용한다. 객체마다 고유한 데이터 키(DEK)를 랜덤 생성하고, 그 DEK를 상위 키로 감싸서 메타데이터에 함께 저장하는 방식이다.

### SSE-S3
S3가 키의 생성, 암호화, 회전을 전부 관리하는 방식이다. 2023년 1월부터 모든 신규 객체에 기본 적용된다.

객체 업로드 시 S3가 내부적으로 랜덤 DEK를 생성하고, AES-256-GCM으로 객체를 암호화한 뒤, S3 루트 키로 DEK를 감싸서 메타데이터에 저장한다. 평문 DEK는 즉시 폐기된다.

추가 비용이 없고 성능 영향도 없다는 것이 장점이지만, 키에 대한 통제권이 전혀 없다. 누가 복호화했는지 감사 로그를 남길 수 없고, 키를 강제로 무효화하거나 회전 주기를 설정할 수도 없다. 모든 것이 AWS 블랙박스 안에서 처리된다.

### SSE-KMS
AWS KMS(Key Management Service)를 통해 고객이 키를 관리하는 방식이다. 봉투 암호화 구조는 SSE-S3와 동일하지만, DEK를 감싸는 상위 키가 고객이 제어할 수 있는 CMK(Customer Master Key)라는 점이 다르다.

객체 업로드 시 S3가 KMS에 GenerateDataKey를 요청하면, KMS가 랜덤 DEK를 생성하고 고객의 CMK로 암호화하여 반환한다. S3는 평문 DEK로 객체를 암호화한 뒤, 암호화된 DEK를 메타데이터에 저장하고 평문 DEK를 폐기한다.

KMS API 호출 비용이 발생하고 처리량 제한이 있지만, 핵심 이점은 **통제와 감사**에 있다. CloudTrail에 모든 복호화 요청이 기록되고, CMK를 비활성화하면 해당 키로 암호화된 모든 객체에 즉시 접근이 차단된다. 키 회전 정책도 고객이 설정할 수 있다. 이러한 이유로 금융, 의료 등 감사 증적이 필요한 규제 산업에서는 SSE-KMS가 요구된다.

### DSSE-KMS
SSE-KMS를 두 겹으로 적용하는 이중 계층 암호화 방식이다. 첫 번째 계층에서 KMS 데이터 키로 객체를 암호화하고, 두 번째 계층에서 S3 관리형 키로 한 번 더 암호화한다. 두 계층이 독립적인 키를 사용하므로 한쪽이 뚫려도 다른 쪽이 데이터를 보호한다. CloudTrail 감사와 키 제어는 SSE-KMS와 동일하게 지원된다. 비용과 성능 오버헤드가 가장 크기 때문에, 다중 계층 암호화를 규정으로 의무화하는 산업(국방, PCI-DSS 레벨 3 등)에서 주로 사용된다.

### SSE-C
고객이 직접 생성한 암호화 키를 매 요청마다 HTTPS 헤더로 전달하는 방식이다. S3는 전달받은 키로 객체를 암호화한 뒤 키 자체는 저장하지 않고 해시값만 보관한다. 복호화 시에도 동일한 키를 다시 전달해야 하며, 해시값으로 검증한 뒤 복호화를 수행한다. AWS가 키를 보관하지 않으므로 키를 분실하면 데이터를 복구할 방법이 없고, CloudTrail에도 기록되지 않는다. 2026년 4월부터 새 버킷에서 기본 비활성화되므로 신규 프로젝트에서는 권장되지 않는다.
