# KMS
![](https://i.imgur.com/XtOrFjG.png)

AWS Key Management Service(KMS)는 데이터를 암호화하는 데 사용되는 암호화 키를 생성하고 관리하는 서비스이다. 

암호화에는 반드시 키가 필요한데, 이 키를 코드에 하드코딩하거나 서버 파일에 저장하면 유출 위험이 크다. KMS는 이 문제를 해결하기 위해 키 자체를 AWS의 전용 하드웨어 보안 모듈(HSM)에 보관하고, 필요할 때만 암호화/복호화 요청을 보내는 방식으로 동작한다. KMS의 HSM은 FIPS 140-3 Level 3 인증을 받은 장비이며, KMS 키의 원본 키 재료(key material)는 이 HSM 밖으로 절대 나가지 않는다.

## 봉투 암호화
KMS의 핵심 아이디어는 "키로 키를 감싸는" 봉투 암호화(Envelope Encryption)이다. KMS 키(마스터 키)로 직접 데이터를 암호화하는 것이 아니라, 데이터 키를 생성한 뒤 그 데이터 키로 실제 데이터를 암호화하고, 데이터 키 자체는 KMS 키로 암호화하여 보관하는 구조이다.

암호화 과정은 다음과 같다. 먼저 KMS에 GenerateDataKey를 요청하면, KMS는 평문 데이터 키와 암호화된 데이터 키 두 가지를 반환한다. 애플리케이션은 평문 데이터 키로 데이터를 암호화한 뒤, 평문 데이터 키를 메모리에서 즉시 삭제한다. 이후에는 암호화된 데이터 키와 암호화된 데이터만 저장한다.

복호화 과정은 그 반대이다. 저장해둔 암호화된 데이터 키를 KMS에 보내면, KMS가 이를 복호화하여 평문 데이터 키를 반환한다. 이 평문 데이터 키로 데이터를 복호화한 뒤, 다시 평문 데이터 키를 삭제한다.

이 구조에서 사용처(애플리케이션)는 마스터 키의 원본을 전혀 알 수 없다. KMS에는 데이터 키의 생성과 복호화 요청만 보낼 뿐, 실제 데이터 암복호화는 로컬에서 수행된다. 따라서 서버가 해킹되어 암호화된 데이터 키와 암호화된 데이터가 모두 유출되더라도, KMS 키에 대한 IAM 권한이 없으면 복호화가 불가능하다.

## KMS 키의 종류
KMS 키는 관리 주체에 따라 세 가지로 나뉜다.

고객 관리형 키(Customer Managed Key)는 사용자가 직접 생성하고 관리하는 키이다. 키 정책, IAM 정책, 권한 부여(Grant) 등을 통해 완전한 제어가 가능하며, 활성화/비활성화, 자동 회전 설정, 삭제 예약 등을 할 수 있다. CloudTrail을 통한 사용 감사도 가능하다. 월별 요금과 사용량 기반 요금이 발생한다.

AWS 관리형 키(AWS Managed Key)는 AWS 서비스가 사용자 계정에 자동으로 생성하는 키이다. alias/aws/s3, alias/aws/ebs와 같이 aws/ 접두사가 붙은 별칭을 가진다. 속성 변경이 불가능하고 해당 AWS 서비스의 컨텍스트에서만 사용할 수 있다. 연 1회 자동 회전이 필수이며, 월별 요금은 없고 사용 요금만 발생한다.

AWS 소유 키(AWS Owned Key)는 AWS 서비스가 내부적으로 소유하고 관리하는 다중 계정용 키이다. 사용자 계정에 존재하지 않으며, 메타데이터 조회나 감사가 불가능하다. 완전 무료이며 별도의 설정 없이 기본적으로 사용된다.

## 저장 데이터 암호화 (Encryption at Rest)
[[AWS S3]]나 RDS 같은 서비스에서 KMS를 통한 암호화는 저장 데이터 암호화(Encryption at Rest), 즉 스토리지 레벨의 암호화를 의미한다. 데이터가 디스크에 쓰여지는 시점에 암호화되고, 읽을 때 자동으로 복호화된다. 애플리케이션 입장에서는 투명(transparent)하게 동작하여 별도의 암호화 코드가 필요 없다.

S3의 경우, 객체를 업로드하면 S3가 내부적으로 KMS에 데이터 키를 요청하여 암호화하고, 다운로드할 때는 KMS에 복호화를 요청하여 평문 데이터를 반환한다. 암호화된 데이터 키는 S3 객체의 메타데이터에 함께 저장되므로, 다운로드 시 S3가 이를 꺼내어 KMS에 복호화를 요청하는 과정이 자동으로 이루어진다.

RDS의 경우, KMS 암호화를 활성화하면 DB 인스턴스의 EBS 볼륨, 자동 백업, 스냅샷, Read Replica, 로그가 모두 디스크 레벨에서 암호화된다. DB 엔진 내부의 쿼리 결과나 메모리 상의 데이터가 암호화되는 것이 아니라 물리적 스토리지에 기록되는 데이터가 암호화되는 것이다. 또한 RDS 인스턴스에 적용된 KMS 키는 직접 변경할 수 없으며, 변경이 필요한 경우 스냅샷을 생성한 뒤 새 KMS 키로 복사하고 해당 스냅샷에서 새 인스턴스를 복원하는 방식을 사용해야 한다.

이러한 저장 데이터 암호화는 디스크 탈취, EBS 스냅샷 유출 등 물리적 접근 위협으로부터 데이터를 보호한다. 다만 IAM 권한이 있는 사용자의 정상적인 데이터 조회나 SQL Injection 같은 애플리케이션 레벨 공격, 네트워크 도청(이는 TLS가 담당)은 방어 범위에 포함되지 않는다.

## CloudTrail을 통한 감사
CloudTrail은 AWS 계정 생성 시 자동으로 활성화되며, KMS에 대한 관리 이벤트(CreateKey, EnableKey, DisableKey, ScheduleKeyDeletion 등)는 별도 설정 없이 기본적으로 90일간 기록된다. AWS 콘솔의 CloudTrail 이벤트 기록에서 이벤트 소스를 kms.amazonaws.com으로 필터링하면 조회할 수 있다.

다만 Encrypt, Decrypt, GenerateDataKey 같은 데이터 이벤트는 기본적으로 콘솔의 이벤트 기록에 표시되지 않는다. 이런 작업은 빈도가 매우 높을 수 있기 때문이다(예: S3 다운로드마다 Decrypt가 호출됨). 데이터 이벤트까지 기록하려면 CloudTrail 추적(Trail)을 별도로 생성하고 KMS 데이터 이벤트 로깅을 활성화해야 하며, 이 경우 S3 저장 비용과 CloudTrail 데이터 이벤트 요금이 발생한다.

## 키 자동 회전
KMS 키의 자동 회전은 암호화 알고리즘을 바꾸는 것이 아니라, 키 재료(key material)만 새 것으로 교체하는 것이다. 키 ID는 그대로 유지되며, 새로 암호화할 때는 최신 키 재료를 사용하고, 기존 데이터를 복호화할 때는 해당 데이터를 암호화했던 시점의 키 재료를 KMS가 자동으로 선택한다. 따라서 자동 회전을 활성화하더라도 애플리케이션 코드를 변경할 필요가 없다.

## 키 식별자
KMS 키는 키 ID, 키 ARN, 별칭(Alias), 별칭 ARN 네 가지 방식으로 식별할 수 있다. 다중 리전 키의 경우 키 ID에 mrk- 접두사가 붙는다. 별칭은 사람이 읽기 쉬운 이름으로, alias/my-app-key와 같은 형식이며, AWS 관리형 키는 alias/aws/s3처럼 aws/ 접두사를 사용한다.
