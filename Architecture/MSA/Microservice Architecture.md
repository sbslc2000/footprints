# Microservice Architecture

마이크로서비스 아키텍처(Microservice Architecture, MSA)는 하나의 애플리케이션을 비즈니스 기능 단위의 작은 서비스들로 분리하여 각각을 독립적으로 개발·배포·운영하는 아키텍처 스타일이다. 각 서비스는 자체 프로세스에서 실행되고, 네트워크를 통해 통신한다.

## 필요성

기존 모놀리식(Monolithic) 아키텍처에서는 모든 기능이 하나의 배포 단위로 묶인다. 서비스 규모가 커지면 다음과 같은 문제가 발생한다.

- **배포 병목** — 주문 로직 한 줄을 수정해도 전체 서비스를 재배포해야 한다.
- **스케일링 비효율** — 결제에만 트래픽이 몰려도 전체를 스케일아웃해야 한다.
- **장애 전파** — 하나의 모듈에서 발생한 장애가 전체 서비스 장애로 이어진다.
- **기술 스택 종속** — 전체가 하나의 언어와 프레임워크에 묶인다.
- **팀 간 의존성** — 코드베이스가 커질수록 팀 간 조율 비용이 증가한다.

MSA는 서비스 분리를 통해 이러한 문제들을 해결한다.

## 핵심 특성

MSA를 구성하는 서비스들은 다음과 같은 특성을 갖는다.

**독립 배포**가 가능하다. 각 서비스는 독립적으로 빌드·테스트·배포되며, 한 서비스의 변경이 다른 서비스의 재배포를 요구하지 않는다.

**독립 데이터 저장소**를 소유한다. 각 서비스는 자체 데이터베이스를 갖는다(Database per Service). 서비스 간 데이터 접근은 반드시 API를 통해 이루어진다.

**경량 통신**을 사용한다. 서비스 간 통신은 REST, gRPC 같은 동기 방식이나 Kafka, RabbitMQ 같은 메시지 브로커를 통한 비동기 방식으로 이루어진다.

**비즈니스 도메인 기준으로 분리**된다. 기술 계층(컨트롤러, 서비스, 레포지토리)이 아닌 도메인 경계를 기준으로 서비스를 나눈다.

**분산 거버넌스**를 따른다. 각 팀이 자기 서비스의 기술 스택을 자율적으로 선택할 수 있다.

## 아키텍처 구성 요소

MSA 환경에서는 서비스 분리로 인해 발생하는 문제들을 해결하기 위한 인프라 구성 요소들이 필요하다.

**API Gateway**는 클라이언트의 요청을 적절한 서비스로 라우팅하는 단일 진입점이다. 인증/인가, 속도 제한, 로드 밸런싱 등의 횡단 관심사를 처리한다.

**서비스 디스커버리**(Service Discovery)는 서비스 인스턴스의 네트워크 위치를 동적으로 파악하는 메커니즘이다. 서비스가 스케일아웃되거나 재시작될 때 IP와 포트가 바뀔 수 있기 때문에 Eureka, Consul 같은 도구를 통해 이를 관리한다.

**분산 데이터 관리**는 서비스별로 분리된 데이터 간의 일관성을 유지하기 위한 전략이다. 분산 트랜잭션을 직접 사용하는 대신 Saga 패턴이나 이벤트 소싱(Event Sourcing)을 통해 최종 일관성(Eventual Consistency)을 달성한다.

**서킷 브레이커**(Circuit Breaker)는 특정 서비스 호출이 반복적으로 실패할 때 해당 호출을 일시적으로 차단하여 장애가 다른 서비스로 전파되는 것을 방지한다.

**모니터링과 관측성**(Observability)은 분산된 서비스들의 상태를 파악하기 위한 체계이다. 분산 트레이싱(Jaeger, Zipkin), 중앙 로깅(ELK Stack), 메트릭 수집(Prometheus, Grafana) 등을 활용한다.
